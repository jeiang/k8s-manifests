# rbac-access

Helm chart for cluster and namespace RBAC.

By default this chart creates:

- `aidanp` as a cluster admin.
- `saeed` as namespace admin in `shared`, `saeed`, and declared namespaces prefixed with `saeed-`.
- `noel` as namespace admin in `shared`, `noel`, and declared namespaces prefixed with `noel-`.
- The namespaces listed above.

Install it with:

```sh
helm upgrade --install rbac-access ./rbac-access \
  --namespace kube-system
```

If the namespaces already exist and are not managed by this chart, set:

```yaml
namespaces:
  create: false
```

Kubernetes RBAC only authorizes identities that your cluster authentication already recognizes. Update `admin.subjects` and each `namespaceAccess.users[].subjects` entry to match the exact usernames, groups, or service accounts from your cluster.

For namespace users, `namespaceAccess.clusterRole` defaults to the built-in `admin` ClusterRole. That grants broad permissions inside the listed namespaces only; it does not grant cluster-wide access.

Kubernetes RBAC does not support namespace wildcard bindings. Prefix access is generated by this chart against the namespaces declared in values. To grant `saeed` access to a future namespace such as `saeed-website`, add it to `namespaces.additional` and run `helm upgrade`:

```yaml
namespaces:
  additional:
    - saeed-website
```

To add another namespace-scoped user, add a new entry under `namespaceAccess.users`:

```yaml
namespaceAccess:
  users:
    - name: example
      subjects:
        - kind: User
          name: example
          apiGroup: rbac.authorization.k8s.io
      namespaces:
        - shared
        - example
```
