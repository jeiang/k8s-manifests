# Chart Upgrade & Risk Review

Generated by a full-repo review of every chart, values-only upstream chart
directory, and plain-manifest directory in this repository, cross-checked
against current upstream documentation, chart repos, and release notes.
`hath/` is excluded per review scope.

Every local Helm chart in scope lints and templates cleanly (`helm lint` /
`helm template`), and every values-only upstream chart directory renders
cleanly against the currently-latest chart in its repo. **Nothing below is a
broken deployment** ‚Äî these are upgrade, hardening, and reliability
recommendations found by comparing this repo's state against current
upstream docs/releases, not failures surfaced by validation.

## Executive Summary

Three cross-cutting patterns account for most of the findings:

1. **Unpinned upstream chart/image versions.** Nearly every values-only
   upstream chart directory (`actual-budget`, `crowdsec`, `monitoring`,
   `rclone-csi-driver`, `netbird-resources`'s operator install) documents
   `helm upgrade --install` with no `--version` flag, and the foundational
   Hetzner CCM/CSI components in `docs/CLUSTER.md` are unpinned too. Every
   install/upgrade run silently tracks upstream's current latest. This is
   fine until it isn't ‚Äî several of these upstreams have shipped breaking
   changes recently (see DOC-1, NR-1).
2. **Single points of failure sit on the public ingress/WAF/DNS path.**
   CrowdSec AppSec runs one replica on a `Recreate` strategy while Traefik's
   bouncer plugin is configured fail-closed globally (CS-TR-1) ‚Äî a routine
   AppSec restart can take the whole site down. Blocky DNS previously OOM'd
   for the same "single replica, expensive cold start" shape (fixed this
   session; see below).
3. **Two self-authored app charts (`bill-splitter`, `website`) are missing
   baseline hardening** (securityContext, liveness/readiness probes) that
   every other chart in this repo already has, and `bill-splitter` additionally
   floats on `image.tag: latest`.

The single highest-impact finding is **MO-2: Alertmanager ships with the
chart's default `blackhole` receiver** ‚Äî this cluster is already generating
a full set of Kubernetes/node alerting rules, and every one of them is
silently discarded. Rules firing with no delivery path is effectively no
alerting at all.

### Already resolved this session (context, not open findings)

- **blocky-dns OOM on startup**: memory reverted from an over-aggressive
  `192Mi` limit to `350Mi`, plus a `startupProbe` added, after the pod was
  OOMKilled while parsing 5 remote denylists before binding port 53.
- **blocky-dns NetBird integration**: `netbird.enabled` flipped to `true` by
  default so the `NetworkResource` renders without an explicit `--set`.
- **netbird STUN outage**: root-caused to stale `AAAA` DNS records for
  `stun.netbird.jeiang.dev` pointing at two old relay nodes' IPv6 blocks
  after the `netbird.io/stun` label moved to `legion-node5`; fixed via DNS
  (outside this repo), with a note added to `netbird/AGENTS.md` to update
  both record types together on future moves.

These are mentioned only so the findings below aren't re-litigated; no
action is needed on them.

## Detailed Findings

### actual-budget

#### AB-1 ‚Äî PVC migration docs contradict current `values.yaml` (High)

- **Evidence**: `actual-budget/values.yaml` has no `persistence.existingClaim`
  key (removed in commit `a3f8529`, "chore: create pvc for actual budget
  chart"). `helm template actual-budget community-charts/actualbudget -f
  actual-budget/values.yaml` (chart `v1.9.1`, current latest in the added
  `community-charts` repo) renders a dynamically-provisioned PVC named
  `actual-budget-data`, not `actual-budget-hcloud`. But
  [`actual-budget/README.md`](actual-budget/README.md) and
  [`actual-budget/AGENTS.md`](actual-budget/AGENTS.md) still describe the old
  workflow: "A pre-created Hetzner CSI-backed persistent volume claim named
  `actual-budget-hcloud`" and an "Existing Data Migration" section
  instructing readers to create that claim and copy data into it.
- **Doc reference**: https://community-charts.github.io/docs/charts/actualbudget/usage
  and `helm show values community-charts/actualbudget` confirm
  `persistence.existingClaim` (default `""`) is still the correct field name
  for pinning to a pre-created claim ‚Äî leaving it unset (as this repo now
  does) makes the chart dynamically provision its own PVC.
- **Why it matters**: This is the exact same drift class that was just fixed
  in `hath/` this session (chart moved from "mount a pre-created PVC" to
  "chart creates its own PVC," but the docs weren't updated). A future
  operator (or agent) following the stale migration steps would pre-create
  an unused PVC or believe a manual data copy is required when it isn't.
- **Implementation**: Update `actual-budget/README.md` and
  `actual-budget/AGENTS.md` to match the pattern already used in
  `hath/README.md`/`hath/VALUES.md`: describe the chart-created
  `actual-budget-data` PVC on `hcloud-volumes`, replace the "copy old PVC
  into `actual-budget-hcloud`" section with "restore from local backup into
  the new chart-managed PVC," and fix the "Dependencies" and "Verify"
  sections' references to `actual-budget-hcloud`.
- **Expected impact**: Docs and values stay consistent; removes risk of
  wasted manual PVC pre-creation or an unnecessary migration attempt.
- **Validation**: `helm template actual-budget community-charts/actualbudget --namespace actual-budget -f ./actual-budget/values.yaml | grep -A5 "kind: PersistentVolumeClaim"` (non-mutating).

#### AB-2 ‚Äî Chart version unpinned in install docs (Medium)

- **Evidence**: `actual-budget/README.md`'s install commands run `helm
  upgrade --install actual-budget community-charts/actualbudget ... -f
  ./actual-budget/values.yaml --wait` with no `--version`. `helm search repo
  community-charts/actualbudget --versions` shows `1.9.1`, `1.9.0`, `1.8.9`,
  `1.8.8` all present locally ‚Äî the install is not deterministic across
  time/machines.
- **Doc reference**: https://github.com/community-charts/helm-charts ‚Äî
  chart versioning is independent of appVersion; `1.9.1` currently maps to
  `actualbudget/actual-server:26.7.0`.
- **Why it matters**: Other charts in this repo (`bitwarden-sm-operator`)
  already record the checked chart version for reproducibility; this one
  doesn't, inconsistent with the repo's own convention.
- **Implementation**: Add `--version 1.9.1` (or whatever is current at edit
  time) to the `helm template`/`helm upgrade --install` commands, and add a
  note recording the chart/app version pair the same way
  `bitwarden-sm-operator/README.md` does.
- **Expected impact**: Upgrades become an explicit, reviewable diff.
- **Validation**: `helm search repo community-charts/actualbudget --versions | head -5`.

### bill-splitter

#### BS-1 ‚Äî Floating `latest` image tag contradicts the chart's own guidance (High)

- **Evidence**: `bill-splitter/values.yaml` sets `image.tag: latest` for
  `ghcr.io/jeiang/bill-splitter`; `bill-splitter/Chart.yaml` has
  `appVersion: "latest"`. This directly contradicts
  [`bill-splitter/AGENTS.md`](bill-splitter/AGENTS.md)'s own editing note:
  "Keep image tag changes explicit and reviewable" ‚Äî `latest` is the
  opposite. Sibling chart `website/` (same author/shape) already uses an
  immutable tag (`sha-6ac94e3`), proving the pattern is achievable here.
- **Doc reference**: General Kubernetes/Helm guidance against `:latest` in
  production (https://kubernetes.io/docs/concepts/containers/images/#image-names)
  ‚Äî no upstream docs apply since this is a self-hosted app.
- **Why it matters**: With `latest` + `imagePullPolicy: IfNotPresent`
  (bill-splitter's default), a node with a cached `latest` layer may never
  pull a newer build at all, causing silent version skew between
  replicas/nodes; forcing `Always` instead makes every rollout
  non-reproducible and undiffable.
- **Implementation**: Pin `image.tag` to an immutable reference ‚Äî a content
  digest or a build-specific tag matching `website`'s `sha-<short-sha>`
  convention, sourced from whatever CI publishes
  `ghcr.io/jeiang/bill-splitter`. Update `Chart.yaml`'s `appVersion` to
  match.
- **Expected impact**: Reproducible, diffable deployments; rollback becomes
  a one-line git revert.
- **Validation**: `helm template bill-splitter ./bill-splitter --namespace bill-splitter | grep image:`; `helm lint ./bill-splitter`.

#### BS-2 ‚Äî No securityContext or liveness/readiness probes (Medium)

- **Evidence**: `bill-splitter/templates/deployment.yaml` sets only `image`,
  `imagePullPolicy`, `ports`, and `resources` on the container ‚Äî no
  `securityContext` and no probes at all. Every other chart reviewed this
  session (`actual-budget`, `blocky-dns`, `hath`) sets an explicit
  `securityContext` (`allowPrivilegeEscalation: false`,
  `capabilities.drop: [ALL]`, `runAsNonRoot: true`) and probes. Sibling
  chart `website/` has the identical gap (see WS-1) ‚Äî this is a shared
  authoring gap across both of this author's self-authored app charts.
- **Doc reference**: Kubernetes Pod Security Standards
  (https://kubernetes.io/docs/concepts/security/pod-security-standards/)
  and probe docs
  (https://kubernetes.io/docs/tasks/configure-pod-container/configure-liveness-readiness-startup-probes/).
- **Why it matters**: Without a readinessProbe, Traefik/Kubernetes routes
  traffic to a pod before it's actually ready; without a livenessProbe, a
  wedged process is never restarted; without a securityContext, the
  container gets more privilege than a simple app server needs by default.
- **Implementation**: Add container-level `securityContext`
  (`allowPrivilegeEscalation: false`, `capabilities.drop: [ALL]`,
  `runAsNonRoot: true`, `readOnlyRootFilesystem` if the image supports it)
  plus `livenessProbe`/`readinessProbe` with `httpGet: {path: /, port:
  http}`, gated behind new `values.yaml` keys the way `actual-budget` does.
- **Expected impact**: Faster detection of a wedged container, no traffic to
  not-yet-ready pods during rollout, reduced blast radius if the image is
  ever compromised.
- **Validation**: `helm template bill-splitter ./bill-splitter --namespace bill-splitter | grep -A10 "livenessProbe\|securityContext"`.

### bitwarden-sm-operator

#### BW-1 ‚Äî Metrics service exists but nothing scrapes it (Medium)

- **Evidence**: `bitwarden-sm-operator/values.yaml` defines `metricsService`

  `crowdsec/crowdsec-vmservicescrape.yaml`'s shape: selector matching the
  sm-operator manager Service's labels, endpoint `port: https, path:
  /metrics, scheme: https` (verify TLS/kube-rbac-proxy behavior with a
  port-forward + curl before finalizing).
- **Expected impact**: Operator sync health becomes visible in the existing
  Grafana/VictoriaMetrics stack, enabling alerting on sync failures.
- **Validation**: `grep -rl "VMServiceScrape" .` (confirms the gap);
  `kubectl apply --dry-run=client -f <new-manifest>.yaml` after adding it.

### blocky-dns

#### BD-1 ‚Äî Upgrade to current Blocky release for on-disk blocklist caching (High)

- **Evidence**: `blocky-dns/Chart.yaml` / `values.yaml` pin `v0.28.2`. Every
  pod (re)start re-downloads and re-parses all 5 remote denylists in memory
  before binding port 53 ‚Äî the exact mechanism that caused this session's
  OOMKill (already fixed by raising memory to 350Mi + adding a
  startupProbe, but the underlying cost is still there on every cold start).
- **Doc reference**: https://github.com/0xERR0R/blocky/releases ‚Äî latest is
  `v0.33.0` (2026-06-30). `v0.31.0` adds an on-disk download cache for
  blocklists with conditional revalidation (skips re-downloading/re-parsing
  unchanged lists on restart); `v0.32.0` adds DNS rebinding protection;
  `v0.33.0` adds a lock-free grouped cache and allocation-free resolver
  selection (lower steady-state memory/CPU).
- **Why it matters**: The on-disk cache attacks the root cause of the OOM
  incident directly (cold-start re-parse cost) rather than just adding
  memory headroom around it. DNS rebinding protection is directly relevant
  for a resolver serving internal/split-horizon queries.
- **Implementation**: Bump `image.tag`/`appVersion` to `v0.33.0`. Check the
  new config schema for a download-cache directory/path key and mount a
  writable volume for it if v0.31+ requires one. Re-verify the
  `startupProbe`'s `failureThreshold` is still generous enough for the very
  first cold start (no cache populated yet).
- **Expected impact**: Faster/more reliable restarts, lower peak startup
  memory, reduced recurrence risk of the OOM class of incident, plus DNS
  rebinding protection.
- **Validation**: `helm template ./blocky-dns --set image.tag=v0.33.0`;
  `helm lint ./blocky-dns`; after a real change, watch `kubectl -n dns logs
  -f deploy/blocky-dns` through a cold restart and confirm memory stays
  well under the 350Mi limit.

#### BD-2 ‚Äî Pin Hagezi CDN URLs instead of `@latest` (Medium)

- **Evidence**: `blocky-dns/values.yaml` uses
  `https://cdn.jsdelivr.net/gh/hagezi/dns-blocklists@latest/hosts/pro.plus.txt`
  and `.../tif.txt`.
- **Doc reference**: https://github.com/hagezi/dns-blocklists ‚Äî jsDelivr's
  own convention (`@latest` vs `@<tag>` vs `@<sha>`) is the pinning
  mechanism; hagezi ships tagged releases jsDelivr can address directly. No
  explicit pinning recommendation found in hagezi's own FAQ, so this is
  general reproducibility practice, not a documented upstream requirement.
- **Why it matters**: `@latest` means every restart or scheduled refresh
  silently picks up whatever is on `main` at that instant ‚Äî no local
  record of which list version is deployed, no rollback, no review step
  before a cluster-wide DNS behavior change takes effect.
- **Implementation**: Replace `@latest` with a pinned tag or commit SHA for
  both hagezi URLs; update deliberately (e.g. quarterly, or via a
  Renovate/Dependabot-style bump). Consider the same for StevenBlack's URLs
  if that repo tags releases.
- **Expected impact**: A bad upstream blocklist push becomes a reviewable
  update instead of an immediate incident.
- **Validation**: `curl -sI https://cdn.jsdelivr.net/gh/hagezi/dns-blocklists@<tag>/hosts/pro.plus.txt`;
  `helm template ./blocky-dns` to confirm the ConfigMap picks up the new URL.

#### BD-3 ‚Äî Remove HPA; use a fixed low replica count instead (Medium)

- **Evidence**: `blocky-dns/templates/hpa.yaml` + `values.yaml`
  (`autoscaling.enabled: true`, CPU 70%, 1-5 replicas) behind a `ClusterIP`
  Service.
- **Why it matters**: Blocky's CPU cost is almost entirely front-loaded at
  startup (parsing lists); steady-state query serving at homelab volumes
  will essentially never hit 70% CPU, so the HPA's trigger is dead weight.
  Worse, when it *did* scale during the prior OOM incident, each new
  replica paid the same memory-hungry startup cost independently
  (multiplying blast radius) with zero cache-sharing benefit across
  replicas (each Blocky pod keeps its own independent in-memory cache).
- **Implementation**: Set `autoscaling.enabled: false` (the HPA template is
  already conditional, no template change needed) and set `replicaCount: 2`
  for availability across node reboots/maintenance, instead of
  CPU-triggered autoscaling.
- **Expected impact**: Removes an unnecessary autoscaler + metrics-server
  dependency and the multiplying-OOM-blast-radius scenario, while keeping
  availability via a fixed 2-replica floor.
- **Validation**: `helm template ./blocky-dns --set autoscaling.enabled=false --set replicaCount=2 | grep -A5 "kind: Deployment"`;
  `helm template ./blocky-dns --set autoscaling.enabled=false | grep -c "kind: HorizontalPodAutoscaler"` should print `0`.

### crowdsec & traefik

These two are reviewed together ‚Äî Traefik loads a CrowdSec bouncer plugin
and forwards to CrowdSec's AppSec/LAPI services, so the biggest finding
spans both.

#### CS-TR-1 ‚Äî AppSec fail-closed + single replica + `Recreate` = full-site outage window on every restart (High)

- **Evidence**: `crowdsec/values.yaml` sets `appsec.replicas: 1` and
  `lapi.replicas: 1`, both on `hcloud-volumes` (RWO) PVCs. `helm template
  crowdsec crowdsec/crowdsec -f crowdsec/values.yaml --namespace crowdsec`
  confirms both the `crowdsec-lapi` and `crowdsec-appsec` Deployments render
  `strategy: {type: Recreate}` (required, since RWO Hetzner volumes can't
  attach to two pods at once). Meanwhile `crowdsec/README.md`'s documented
  Traefik dynamic config sets `crowdsecAppsecFailureBlock: true` and
  `crowdsecAppsecUnreachableBlock: true` (fail-closed), and
  `traefik/traefik-helmchartconfig.yaml` applies `crowdsec-bouncer@file`
  globally to **every** route on both the `web` and `websecure` entrypoints
  ‚Äî no per-route bypass.
- **Doc reference**: https://github.com/maxlerebourg/crowdsec-bouncer-traefik-plugin ‚Äî
  `crowdsecAppsecUnreachableBlock`/`crowdsecAppsecFailureBlock` both default
  `true` (block on failure/unreachable).
- **Why it matters**: Because the bouncer is wired globally and fails
  closed, *any* AppSec pod restart (chart bump, node drain, OOMKill,
  routine node reboot) blocks all HTTP/HTTPS traffic through Traefik
  cluster-wide for the entire Recreate cycle (terminate ‚Üí volume detach ‚Üí
  reschedule ‚Üí volume attach ‚Üí ready) ‚Äî not just AppSec-inspected paths.
  This is a single point of failure for the whole ingress path, and it's
  easy to trigger accidentally (e.g. bumping `appsec.resources` or the
  chart version).
- **Implementation**: Pick one: **(a)** run `appsec.replicas: 2` with pod
  anti-affinity so a rolling upgrade never drops to zero ready pods
  (AppSec pods are likely stateless enough to drop the per-replica RWO PVC
  assumption); **(b)** relax `crowdsecAppsecUnreachableBlock`/
  `crowdsecAppsecFailureBlock` to `false` (fail-open) so a brief AppSec
  outage degrades WAF coverage instead of taking the site down; **(c)**
  keep fail-closed only for routes that need it and fail-open elsewhere via
  a second, narrower middleware. Document whichever tradeoff is chosen in
  `crowdsec/AGENTS.md`.
- **Expected impact**: Removes (or makes an explicit, reviewed tradeoff for)
  a full-site-down failure mode tied to routine CrowdSec maintenance.
- **Validation**: `helm template crowdsec crowdsec/crowdsec --namespace crowdsec -f crowdsec/values.yaml | grep -A3 'strategy:'`;
  for a real test, `kubectl -n crowdsec rollout restart deploy/crowdsec-appsec`
  during a maintenance window while timing `curl -o /dev/null -s -w '%{http_code}\n' https://<any-ingress-host>`.

#### TR-1 ‚Äî CrowdSec bouncer plugin pinned to v1.3.3, a security fix behind (Medium)

- **Evidence**: `traefik/traefik-helmchartconfig.yaml` pins `version:
  v1.3.3` for `github.com/maxlerebourg/crowdsec-bouncer-traefik-plugin`.
- **Doc reference**: https://github.com/maxlerebourg/crowdsec-bouncer-traefik-plugin/releases ‚Äî
  latest stable is `v1.6.0` (a `v1.7.0-alpha` exists, avoid it). `v1.6.0`'s
  notes describe a fix for an "AppSec bypass vulnerability related to
  invalid content-length headers" ‚Äî directly relevant since this cluster
  runs `crowdsecMode: appsec`. Intermediate releases add self-hosted
  CAPTCHA support and separate TLS config for LAPI vs AppSec.
  `v1.7.0-alpha` renames `BanHTMLFilePath`‚Üí`BanFilePath` and
  `CaptchaHTMLFilePath`‚Üí`CaptchaFilePath` (not used here, so irrelevant
  unless upgrading past v1.6.0 later).
- **Why it matters**: Since AppSec is the primary WAF path here (dedicated
  bouncer key, `crowdsecMode: appsec`), an AppSec bypass fix landing three-

### website

#### WS-1 ‚Äî No securityContext or liveness/readiness probes (Medium)

- **Evidence**: `website/templates/deployment.yaml` (25 lines total) sets
  only `image`, `imagePullPolicy`, `ports`, `resources` ‚Äî no
  `securityContext`, no probes. Same structural gap as `bill-splitter`
  (BS-2), but `website` is more exposed since it's the only chart in this
  review actually serving public production traffic across three domains
  (`jeiang.dev`, `aidanpinard.co`, `pinard.co.tt`) with `redirectToHttps:
  true` already configured.
- **Doc reference**: General Kubernetes best practice
  (https://kubernetes.io/docs/concepts/workloads/pods/pod-lifecycle/#container-probes,
  Pod Security Standards restricted profile) ‚Äî no single upstream doc since
  this is self-authored.
- **Why it matters**: Without a readinessProbe, Kubernetes marks the pod
  ready as soon as the process starts, before the web server is actually
  accepting connections ‚Äî causing brief 502s during rollouts. Without a
  livenessProbe, a wedged container is never auto-restarted. Without a
  securityContext, the container runs with more privilege than a
  static-file server needs.
- **Implementation**:
  ```yaml
          securityContext:
            runAsNonRoot: true
            readOnlyRootFilesystem: true
            allowPrivilegeEscalation: false
            capabilities:
              drop: ["ALL"]
          readinessProbe:
            httpGet:
              path: /
              port: http
            initialDelaySeconds: 2
            periodSeconds: 10
          livenessProbe:
            httpGet:
              path: /
              port: http
            initialDelaySeconds: 5
            periodSeconds: 20
  ```
  Verify the base image can run as non-root/read-only-rootfs (nginx/caddy
  typically need a writable path for temp/cache ‚Äî may need `emptyDir`
  mounts); adjust the probe path if the image doesn't serve `200` at `/`.
- **Expected impact**: Eliminates brief unready-pod traffic during
  deploys/restarts, auto-recovers from a hung container, reduces container
  privilege surface for a public-facing service.
- **Validation**: `helm lint ./website`; `helm template test ./website --namespace website | grep -A5 -iE "probe|securityContext"`.
